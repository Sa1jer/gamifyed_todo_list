# Product Doctrine v1

## Product purpose

**Product principle:** RPG To-Do exists to help a person turn an important
intention into one concrete, finishable action without requiring them to
reconstruct the whole project in their head.

The RPG layer is feedback and language for real-world movement, not the reason
to perform empty work.

## Core user problem

**Product hypothesis:** the central problem is not insufficient task storage.
It is the gap between "I know this matters" and "I know what I can do now,
what counts as enough, and how to return later." That gap may contain
ambiguity, activation energy, context loss, perfectionism, decision overload,
or emotional resistance.

## Product promise

The product should consistently help a user:

> See one useful next action, make a bounded real-world move, record enough
> context to return without punishment, and recognise evidence of progress.

It may suggest structure; it must not claim certainty about the user's mind,
health, capacity, or priorities.

## Four failure states

| State | Common experience | Helpful intervention classes | Harmful patterns | Success |
| --- | --- | --- | --- | --- |
| START | "I cannot begin." | Explicit next physical action, Boot sequence, reduced scope, context opening, capacity choice. | More setup, generic exhortation, forced reward chase. | The user performs one small observable entry action. |
| CONTINUE | "I began, then lost the thread." | Quest Chain, checkpoint, blocker note, Save Point, visible result. | Asking them to re-plan the whole project mid-session. | The user knows the next bounded move without rediscovery. |
| FINISH | "I do not know when enough is enough." | Definition of Done, scope boundary, finish review, parked improvements. | Infinite polish, reward for volume alone, false urgency. | The result meets an agreed acceptable boundary. |
| RETURN | "I stopped and now it feels expensive." | Gentle return summary, Save Point, small re-entry, backlog quieting. | Streak loss, guilt copy, stale backlog dump. | The user reconnects with context and takes a low-cost next step. |

## Product principles

1. **One useful next action beats a larger menu of worthy actions.**
2. **Reduce decisions before adding motivation.**
3. **Prefer observable outcomes to vague effort.**
4. **Preserve return context before rewarding completion.**
5. **Real progress is more important than virtual progress.**
6. **A pause does not invalidate a journey.**
7. **Support must not become pressure.**
8. **Capacity can vary without changing the user's worth.**
9. **Explicit instructions are kinder than implied expectations.**
10. **The user remains in control of focus, scope, and pacing.**
11. **The product may suggest patterns, never diagnose a person.**
12. **A simpler intervention wins when it creates the same real movement.**

## Product anti-principles

- Productivity shame, streak punishment, loss aversion, and moralised labels.
- XP farming, random rewards, or feature completion mistaken for project work.
- Infinite planning before a user can make a first move.
- Mandatory daily use, one ideal workflow, or compulsory positivity.
- Medical, diagnostic, or treatment claims.
- Hiding crucial context behind game mechanics or advanced configuration.
- Treating research, maintenance, rest, or a pause as automatic failure.

## Product personality

Calm, direct, respectful, specific, and nonjudgmental. The product can say
"this step may be too large" or "you paused here". It should not say "you
failed," "you are behind," or manufacture celebration without real evidence.

## Decision test

Every future feature must answer:

1. Does it make starting, continuing, finishing, or returning easier?
2. Does it lower cognitive load or context reconstruction?
3. Does it make real-world progress clearer?
4. Could it reward avoidance, busywork, or endless planning?
5. Could it increase shame, pressure, or false urgency?
6. Is the user still able to decline or change the suggestion easily?
7. Can an existing primitive such as Task, Minimum Action, Review, or history
   provide the value with less new state?
8. What evidence would show that it does not help or makes the app another
   burden?

If a proposed feature cannot answer these questions, it remains an experiment,
not a core product commitment.
