# RPG Mechanics Reassessment

## Revised philosophy

**Product principle:** real-world action is the game. RPG presentation should
make real work legible, bounded, and satisfying; it must not make the user farm
virtual progress while avoiding the intended outcome.

| Mechanic | Classification | Why | Guardrail |
| --- | --- | --- | --- |
| Quest completion feedback | CORE | Marks a completed work cycle. | Keep completion fast; do not require extra ceremony. |
| Minimum Action | CORE | Existing start accommodation. | Preserve its current XP and undo semantics. |
| XP | SUPPORTING | Makes completed work visible. | Show after real work; do not make XP the first decision. |
| Skill / character levels | SUPPORTING | Offers long-term continuity. | Avoid framing lower level as personal failure. |
| RoadMap stages | SUPPORTING | Shows capability or project path. | Never require RoadMap setup to begin a task. |
| Review | SUPPORTING | Supports reflection and adjustment. | Optional, contextual, and not a productivity grade. |
| Nudge | SUPPORTING | Existing rules can surface missing clarity. | Suggest, do not command; explain the reason. |
| Proof of Progress | CORE candidate | Connects completion to real-world result. | Optional and non-blocking until validated. |
| Save Point | CORE candidate | Makes return easier. | Start derived; add persistence only after validation. |
| Journey framing | SUPPORTING | Counters all-or-nothing streak thinking. | Do not hide deadlines or practical reminders. |
| Achievements | OPTIONAL | Can celebrate meaningful moments. | No mandatory collection loop. |
| Chests / rewards / buffs | OPTIONAL | Novelty after work for users who enjoy it. | Never gate execution or inflate completion pressure. |
| Milestone animations / sound | OPTIONAL | Recognition of real milestones. | Respect reduced motion and sound preferences. |
| Streak-like systems | DEFER | Current recurrence data can be useful, but loss framing conflicts with doctrine. | Reassess copy and value before expanding. |
| Fantasy terminology | OPTIONAL | Can support identity and warmth. | Plain-language alternative must remain understandable. |
| Main / Side Quest distinction | SUPPORTING candidate | Can clarify focus and drift. | Do not require taxonomy until it changes a decision. |
| Boss / resistance event | DEFER | Potentially expressive feedback. | Avoid adversarial or clinical framing; validate separately. |

## Anti-farming test

Before strengthening any mechanic, ask:

1. Can the user gain this reward while their intended real-world result does
   not move?
2. Does it make starting easier, or only make logging activity more tempting?
3. Does it create loss aversion after a pause?
4. Can it be hidden or reduced without impairing the core action loop?

If the answer to the first or third question is yes, keep the mechanic optional
or reduce its visual priority.
