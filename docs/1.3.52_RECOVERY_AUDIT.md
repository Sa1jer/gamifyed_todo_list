# 1.3.52 Recovery Audit

Last updated: 2026-07-05

## Scope and baseline

The audited commit is `40acade5a61fc634ca48099fc7d0e8884a637e6a`
(`1.3.52`). Its parent and resulting behavior were compared with the full
Application-Wide Visual Language Migration Epic. The audit confirmed the
reported failure pattern: several workspace routes existed, but still rendered
legacy composition or incomplete content.

Baseline before recovery was clean: formatter unchanged, analyzer clean, and
`312/312` tests passed. The recovery changes presentation and local debug-tool
gating only. Models, Hive schema, XP formulas, RoadMap graph semantics, and
storage contracts are unchanged.

## Visual verification checkpoints

- **RoadMap:** widget evidence verifies the approved canvas remains, while the
  toolbar, context rail, templates, selection sync, and camera behavior are new.
- **Trophies:** widget evidence verifies all four requested sections and empty
  states.
- **Statistics:** widget evidence verifies the new overview, separated right
  rail, and all eight in-shell detail routes.
- **Skill workspace:** widget evidence verifies the raised header, responsive
  compact mode, history-aware empty state, and fixed action geometry.
- **Quest Creation:** existing mobile and desktop form tests verify order,
  description, XP normalization, settings, Minimum Step, and dirty Back.
- **Settings/Admin:** widget and static security tests verify real setting
  coverage, five-tap behavior, timeout reset, and release guards.

Native screenshot comparison, compositor pointer behavior, and physical Windows
QHD scaling cannot be certified by widget tests. Those checks remain explicitly
open in `TODO.md`.

## Intentional limits

- A skill whose only quest was deleted before it ever produced history cannot
  be distinguished from a never-used skill without a new persisted lifecycle
  field. No schema field was added in this recovery.
- Debug Admin is available in debug and profile builds for developer testing,
  but every entry and mutation/storage boundary rejects release mode.
- Existing mobile/tutorial modal flows remain because the migration target is
  the primary desktop workspace, not every secondary compact flow.
